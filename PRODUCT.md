# Product

## Register

brand

## Users

Recruiters and hiring managers screening for a data analyst internship — scanning quickly, deciding within minutes whether to read further. Secondarily, technically literate reviewers who may actually open project code on GitHub. Vaughn's audience is not casual; they know what SQL, Power BI, and ETL mean.

## Product Purpose

A personal portfolio for Vaughn Lee Edwards Jr. — data analytics and computer science student at James Madison University. The site's job is to make his skills, projects, and experience immediately legible to a recruiter without friction, and credible to a technical reviewer who wants to go deeper. Success means the visitor understands what he can do before they scroll halfway down.

## Brand Personality

Calm, precise, data-native. The site should feel like a sharp internal analytics tool that happens to be about a person — not a startup landing page, not a creative portfolio. No persuasion mechanics. Let the work speak.

## Anti-references

- Sales pages, persuasion funnels, "Hire me!" banners, fake urgency
- Generic SaaS landing pages (rounded hero cards, purple-to-blue gradients, testimonial carousels)
- The standard developer dark-mode portfolio (neon green on black, hacker cliché)
- Stock agency look: large stock photos, lifestyle imagery, lifestyle copy
- Anything with marketing copy voice ("unlock," "supercharge," "transform your")
- Generic résumé-website templates with thin single-column timeline and download link

## Design Principles

1. **Data first, decoration second.** Every visual element should either communicate content or create structure. No ornament for its own sake.
2. **Specificity over superlatives.** Concrete facts (1,000+ records cleaned, 5+ seasons of data, 100+ inventory items tracked) beat adjectives. The design should frame and elevate these specifics.
3. **Consistent register.** The site is an analytics surface, not a marketing surface. The visual language (typefaces, color use, layout density) should match the tools the work was built in — terminal-adjacent, not startup-adjacent.
4. **Quiet confidence.** The site should not understate, but it should never oversell. No progress bars on skills, no fabricated metrics.
5. **Findable depth.** Every project must be fully readable on the page. GitHub links are exits for people who want to go deeper, not the only path to learning what the project is.

## Accessibility & Inclusion

WCAG 2.1 AA minimum. Visible keyboard focus states on all interactive elements. Full `prefers-reduced-motion` support — every animated element must have a static fallback. All interactive elements have accessible labels.
